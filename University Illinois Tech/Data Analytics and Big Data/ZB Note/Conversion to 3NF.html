Conversion to 3NF

Conversion to 3NF

  • A relation will be in 3NF if it is in 2NF and not contain any transitive partial dependency.

  • 3NF is used to reduce the data duplication. It is also used to achieve the data integrity.

  • If there is no transitive dependency for non-prime attributes, then the relation must be in third normal form.

Step 1: Create new tables/relations to remove all transitive dependencies (TD)

For every transitive dependency, write its determinant (JobTitle) as PK for new table. Further, we retain a copy of the determinant in the original table (where TD exists) to serve as a foreign key.

In this case, the original table is:

table2: Employee(EmployeeID, EmployeeName, JobTile, Salary) -- TD exists in this table transitive dependency: JobTitle →\rightarrow→right arrow Salary

Step 2: Reassign corresponding dependent attributes to a PK in the new table

Identify attributes dependent on each determinant identified in Step 1. In this case, Salary is dependent on JobTitle Name the new table to reflect its contents and function (table "Job" reflect JobTitle and Salary).

Therefore new table is:

table5: Job(JobTitle, Salary)

while table2 becomes:

table2: Employee(EmployeeID, EmployeeName, JobTile)

Note: JobTitle is PK in Job table and FK in Employee table.

The remaining tables (from 2NF with no TD) are carried forward without any changes being made. i.e.:

Third normal form relation schemas

In summary, 3NF would consists of 5 tables with the following schemas:

table1: Project(ProjectID, ProjectName)

table3: Department(DepartmentID, DepartmentName)

table4: ProjectEmployee(ProjectID, EmployeeID, DepartmentID, Budget,BudgetedTime)

Third normal form relation schemas

In summary, 3NF would consists of 5 tables with the following schemas:

table1: Project(ProjectID, ProjectName)

table2: Employee(EmployeeID, EmployeeName, JobTile)

table3: Department(DepartmentID, DepartmentName)

table4: ProjectEmployee(ProjectID, EmployeeID, DepartmentID, Budget, BudgetedTime)

table5: Job (JobTitle, Salary)

Note that when a primary key is considered to be unsuitable, designers may use surrogate keys. A surrogate key is an artificially generated unique identifier assigned to each row in a table and may not have any inherent meaning to the data.

In this case, JobTitle is not a suitable PK in Job table (table5). We then introduce JobCode as a PK. Therefore, changes must be made to table2 and table5 as follows.

table2: Employee(EmployeeID, EmployeeName, JobCode)

table5: Job(JobCode, JobTitle, Salary)

Third normal form dependency diagrams

Third normal form relation schemas

table1: Project (ProjectID, ProjectName)

table2: Employee (EmployeeID, EmployeeName, JobCode)

table3: Department (DepartmentID, DepartmentName)

table4: ProjectEmployee (ProjectID, EmployeeID, DepartmentID, Budget, BudgetedTime)

table5: Job (JobCode, JobTitle, Salary)

Third normal form tables with data values

The tables above satisfy the third normal form requirements:

  • Each table is in 2NF.

  • None of the tables include transitive dependencies.
