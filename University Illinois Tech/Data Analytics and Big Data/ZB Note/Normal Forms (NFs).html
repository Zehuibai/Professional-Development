Conversion to 1NF

Step 1: Eliminate the repeating groups.

  • Eliminate nulls: each repeating group attribute contains an appropriate data value. Look for attributes in the relation that contain multiple values within a single tuple.

  • These attributes represent repeating groups and need to be removed.

  • In the example, Project information is a repeating group for each employee.

Step 2: Identify the Primary Key

  • The primary key that uniquely identifies attribute value must be identified.

  • In the example PK: (ProjectID + DepartmentID + EmployeeID).

Step 3: Identify All Dependencies

  • Draw the dependency diagram to depict all dependencies among attributes of the table

  • The arrows above the attributes indicate desirable dependencies (i.e., based on the PK)

  • The arrows below the attributes indicate less desirable dependencies (partial and transitive)

Example: ProjectEmployee Table in Unnormalized Form

Table in First Normal Form

First Normal Form Relation Schemas

ProjectEmployee (ProjectID, ProjectName, EmployeeID, EmployeeName, JobTile, Salary, DepartmentID, Department, Budget, BudgetedTime)

First Normal Form Dependency Diagram

Partial dependencies (PD)

PD1: ProjectID →\rightarrow→right arrow ProjectName

PD2: EmployeeID →\rightarrow→right arrow EmployeeName, JobTile, Salary

PD3: DepartmentID →\rightarrow→right arrow DepartmentName

A combination of all key attributes forms a composite primary key: ProjectID + EmployeeID + DepartmentID; that functional determinant the rest of the attributes in the table.

ProjectID, EmployeeID, DepartmentID →\rightarrow→right arrow Budget, BudgetedTime

Transitive dependencies (TD)

TD1: JobTitle →\rightarrow→right arrow Salary

The table (ProjectEmployee) at First Normal Form (1NF) satisfies the following requirements:

  • No repeating groups in the table

  • The primary key attributes are defined (by identifying all key attributes)

  • All attributes are dependent on primary key.
