 Conversion to 2NF

 Conversion to 2NF

  • The relation must first be in 1NF.

  • The relation is automatically in second normal form (2NF), if the first normal form (1NF) key is a single attribute.

  • If the relation has a composite PK, then each non-key attribute must be fully dependent on the entire PK and not on a subset of the PK (no partial dependency).

Step 1: Create new tables/relations to remove all partial dependencies

For each component of the PK(i.e., ProjectID, EmployeeID, and DepartmentID) that acts as a determinant in a partial dependency, must form a new table with a copy of that component as the PK. In this case, we form three tables:

table1: Project(with ProjectId as PK)

table2: Employee(with EmployeeId as PK)

table3: Department(with DepartmentID as PK)

Note that these components also remain in the original table to serve as FKs to the original table. A combination of these components in the original table forms a composite key, which must be the PK in the original table.

table4: Orginal table (ProjectEmployee -- with ProjectID + EmployeeID + DepartmentID as PK)

Step 2: Reassign corresponding dependent attributes to a PK in each table.

The attributes in a partial dependency are removed from the original table (ProjectEmployee) and placed in the new table with the corresponding dependency’s determinant/key.

For example:

ProjectID →\rightarrow→right arrow ProjectName the determinant (ProjectID) and ProjectName (dependent) become attributes in table1 (Project)

For any attributes that are NOT dependent in a partial dependency remain in the original table. In this case, Budget and BudgeTime not dependent on a partial dependency. These remain in the original table (ProjectEmployee) together with copies of each components of the composite PK (ProjectID + EmployeeID + DepartmentID)

Second normal form dependency diagrams

NOTICE that a transitive dependency (TD1: JobTitle →\rightarrow→right arrow Salary) exists in Employee table.

Second normal form relation schemas

New tables in 2NF are as follows:

table1: Project (ProjectID, ProjectName)

table2: Employee (EmployeeID, EmployeeName, JobTile, Salary)

table3: Department (DepartmentID, DepartmentName)

table4: ProjectEmployee (ProjectID, EmployeeID, DepartmentID, Budget, BudgetedTime)

Second normal form tables/relations with data values

The tables satisfy the second normal form requirements:

  • Each table is in 1NF.

  • None of the tables include partial dependencies.

  • No attribute is dependent on only portion of primary key -- each attribute is fully dependent on a PK in each table.
