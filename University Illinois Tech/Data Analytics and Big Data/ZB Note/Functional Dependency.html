Functional Dependency

Functional dependency theory is a fundamental concept in relational database design that helps ensure data integrity and eliminate data anomalies (such as insertion, deletion and update see more). It provides a set of rules and guidelines for analyzing and representing relationships between attributes in a relational database. It states that for a given relation, the values of one set of attributes (the dependent attributes) are uniquely determined by the values of another set of attributes (the determinant attributes).

In a relational database, functional dependency is represented as follows:

A \rightarrowright arrow B

This notation indicates that attribute set A functionally determines attribute set B. It means that the values of attributes in set B are uniquely determined by the values of attributes in set A (`the relationship from A to B must be "to-one" mapping`)

Formal Definition of a Functional Dependency

Let RRRR be a relation schema

\alphaalpha \subseteq\subseteq RRRR and \betabeta \subseteq\subseteq RRRR

The functional dependency

\alphaalpha \rightarrowright arrow \betabeta

holds on RRRR if, and only if, for any legal relations r(RRRR), whenever any two tuples t1t_1t1t, start subscript, 1, end subscript and t2t_2t2t, start subscript, 2, end subscript of r agree on the attributes \alphaalpha, they also agree on the attributes \betabeta. That is,

t1t_1t1t, start subscript, 1, end subscript[\alphaalpha] = t2t_2t2t, start subscript, 2, end subscript [\alphaalpha] => t1t_1t1t, start subscript, 1, end subscript[\betabeta ] = t2t_2t2t, start subscript, 2, end subscript [\betabeta]

Example: Consider r(A,B ) with the following instance of r.

A B

t1t_1t1t, start subscript, 1, end subscript 1 4

t2t_2t2t, start subscript, 2, end subscript 1 5

t3 t_3t3t, start subscript, 3, end subscript 3 7

On this instance, B \rightarrowright arrow A hold; A \rightarrowright arrow B does NOT hold.

Determinant: The determinant is the set of attributes on the left side of the functional dependency arrow (\rightarrowright arrow). It is the determining factor or condition that uniquely determines the values of the dependent attributes.

Example:

LHS (determinant) -> RHS (dependent)

A \rightarrowright arrow B

LHS = A is determinant (LHS can be more than one attribute)

Dependent: The dependent is the set of attributes on the right side of the functional dependency arrow. These attributes are determined by the values of the determinant attributes.

Example:

LHS (determinant) -> RHS (dependent)

A \rightarrowright arrow B

RHS = B is dependent (RHS can be more than one attribute)

Full Functional Dependency: A functional dependency is considered full if removing any attribute from the determinant would break the dependency. It means that all attributes in the determinant are necessary for determining the values of the dependent attributes.

Example:

LHS (determinant) \rightarrowright arrow RHS (dependent)

AB \rightarrowright arrow C

LHS = both A and B are needed

(C is fully dependent on AB)

Partial Functional Dependency: A functional dependency is partial if it is possible to remove some attributes from the determinant while still maintaining the dependency. In other words, not all attributes in the determinant are necessary to determine the values of the dependent attributes.

Example:

LHS (determinant) -> RHS (dependent)

AB \rightarrowright arrow C, but

B \rightarrowright arrow C

LHS = either A or B can be removed. if A is removed B\rightarrowright arrow C (C is partially dependent on AB)

Transitive Dependency: Transitive dependency occurs when there is a functional dependency between attributes indirectly through other attributes.

Example:

if A\rightarrowright arrow B ( A is prime key, and B is non-prime key)

and B \rightarrowright arrow C

then A \rightarrowright arrow C (transitive dependency)

NOTE: The existence of a functional dependency via a non-prime attribute is a sign of transitive dependency.
